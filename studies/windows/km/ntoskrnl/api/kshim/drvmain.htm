DRVMAIN.SDB

What the Kernel Shim Engine works with as its input data is a Shim Database (SDB) file named DRVMAIN.SDB in the AppPatch subdirectory of the Windows installation. The contents of this Driver Database file are read into memory by the loader and passed to the kernel via the DrvDBImage and DrvDBSize members of the LOADER_PARAMETER_EXTENSION.

Kernel-mode programmers, system administrators and arguably even plain ordinary users have obvious cause to understand the contents of the DRVMAIN.SDB file. If nothing else, computer users have a reasonable interest in knowing what run-time changes Windows makes to the software that’s on their computers or what unusual support has turned out to be needed by software they’re thinking to buy. There’s surely no good reason that this information be obscured from them by burying it in some undocumented file format.

Now, I do accept that consumers, even advanced ones, and even expert kernel-mode programmers, can’t know even with difficulty why some manufacturer’s driver or device appears in an SDB file. For instance, the SDB file that I present below is dated August 2013 and already lists a driver from Kaspersky Anti-Virus as blocked if built as recently as January 2013. Is that because Kaspersky Lab is more likely than others to write a driver that so soon needs attention? Or is it that Kaspersky Lab is merely more diligent about finding and owning the inevitable faults? If the latter, is quality control the motivation or is there some strategic interest in nudging the update cycle by dismissing software as obsolete if it’s not the very latest? If you think this too cynical, then examine successive SDB files and ask why the same drivers don’t just stay listed as broken in this version or if built before that date but instead have advancing link dates and version numbers. As much as I might hope the industry is moving generally in the direction of greater openness, that the listing below appears to be the Internet’s first serious attempt at presenting an SDB file in anything like its original XML suggests that obfuscation is deeply established.

De-Compilation

Here is the DRVMAIN.SDB from the installation disc for 32-bit Windows 8.1, represented as XML. I may extend this to take in 64-bit Windows 8.1 also. I most likely won’t continue this for later versions or track the file’s development through earlier versions.

Please do appreciate that what I present here is not some arbitrary dressing of SDB tags as XML for supposedly easier review on the naive assumption that each TAG in the SDB file corresponds one-to-one with an XML tag or attribute as Microsoft prepares it or as anyone might best read it. If you want to re-compile this XML, if only out of scepticism that this XML really is so much more worth your time than is any other rendering, then unless something goes very wrong for me but not for you when installing the Application Compatibility Toolkit, you may wonder what I mean with this talk of Microsoft’s Shim Database Compiler. When I install the Application Compatibility Toolkit I certainly don’t get a shimdbc.exe program and I doubt that you do either. But it is there, hidden in the code and data of the Compatibility Administrator, and you can “extract” it and run it, e.g., 

shimdbc Driver -op X86 win81x86.xml win81x86.sdb >stdout.txt

For details, see my article Where Is ShimDBC.exe? which is published separately as PoC||GTFO 13:9.

Once you’re satisfied that this XML does indeed re-compile, you may want the tool that produced the XML. Part of me wants to say that you should be able to write it yourself—certainly if you have ever been pleased to present a paper, e.g., at a conference, on your research into SDB files and their rendering into XML. I do recognise, however, that the work of finding what XML to de-compile to is far beyond practicality, not just commercially but even academically, and so I do publish what Shim Database De-Compiler I have. Please allow, however, that even with my long history of over-indulging the plainly impractical, I do not have the resources for being wildly impractical. The de-compiler is therefore severely constrained in its ambition.

Editorial Choices

For any one SDB file, there are of course many XML representations that would all re-compile to that same SDB file (except only for the TAG_TIME that records when the SDB file got compiled). Comments, obviously, are lost by compilation and cannot be recovered. But even for content that translates more or less directly from XML to SDB, there are editorial choices.

Ordering

Attributes within an XML tag can be reordered without affecting the SDB file. Many tags can be too. There is not, by rule, a one-to-one correspondence of SDB tags to XML tags and attributes, but I try to keep to what correspondences can be identified. Consider, for instance, the child tags of the TAG_DATABASE (0x7001) that are themselves list tags. Because Microsoft’s compiler writes them in the order

  • TAG_EXE (0x7007)
  • TAG_LOOKUP (0x7017)
  • TAG_KDRIVER (0x701C)
  • TAG_KDEVICE (0x701A)
  • TAG_KSHIM (0x7025)

my de-compiler writes the “corresponding” XML tags in the order

  • <EXE>
  • <LOOKUP>
  • <KDRIVER>
  • <KDEVICE>
  • <KSHIM>

That much is straightforward, but the ordering within each list is often not. Because the TAG_EXE list in the SDB file is ordered alphabetically by the TAG_NAME (0x6001) child, you might hope that all <EXE> tags in the XML file will be ordered alphabetically by their NAME attribute, but keeping to this ordering exactly is simply not possible. Though all TAG_EXE tags in the SDB file are immediate children of the TAG_DATABASE, the <EXE> tags in the XML file are not direct children of <DATABASE>. Not only must there always be an <APP> tag in between, but where two TAG_EXE tags in the SDB file have the same GUID as data for their TAG_APP_ID (0x9011) tags, the corresponding <EXE> tags in the XML must be under the one <APP> tag that has this GUID for its ID attribute.

Aliases

That the listing above does not have <APP> and <EXE> tags is because Microsoft’s compiler allows aliases for each, and some seem more appropriate for the driver database. Specifically, <APP>, <DRIVER> and <KENGINE> can (mostly) be substituted one for another, as can <EXE> and <SYS> (and <DATA> and <DRIVER_POLICY>, while we’re at it).

The aliasing is close but not exact. For instance, the compiler distinguishes <EXE> and <SYS> such that wildcards are allowed in the NAME attribute for the former but not the latter. Other differences act at a distance. For instance, the BLOCK attribute is allowed in an <APPHELP> only if it is in turn inside an <EXE> and BLOCK_UPGRADE only if inside <SYS>. There is much still to learn, though much of it doesn’t matter immediately (e.g., because the BLOCK and BLOCK_UPGRADE attributes aren’t actually used in the Windows 8.1 DRVMAIN.SDB file). As a possibly unsound editorial choice, I have preferred <DRIVER> everywhere, but perhaps <KENGINE> would be better for the lists of devices. To prefer <SYS> to <EXE> in a driver database seems relatively sound since it’s Microsoft’s own practice for the XML output of the SdbGrabMatchingInfo function in the GRABMI_FILTER_DRIVERS case. Indeed, the round trip of de-compiling and re-compiling a driver database can fail without <SYS> rather than <EXE> under <DRIVER> or <KENGINE> rather than <APP>, depending on other choices, but I cannot say it’s beyond doubt that driver databases must use <SYS> not <EXE>.

Compiler-Generated SDB Tags 

Some tags in the SDB file do not originate in the XML but can only be compiler-generated. Of particular interest are TAG_TIME (0x5001), TAG_COMPILER_VERSION (0x6022) and TAG_OS_PLATFORM (0x4023) as immediate children of the TAG_DATABASE. The DRVMAIN.SDB that is represented above was compiled on 08/21/2013 23:38:29.883 by SHIMDBC version 2.1.0.3 for the X86 platform.

Compiler Versions

Incidentally, if you wonder why in 2016 I have chosen to present a de-compilation of an SDB file from 2013, my answer is that 2.1.0.3 is by far the most widely applicable of all known SHIMDBC versions, having been used for the SDB files on the Windows discs from as far back as Windows Vista until superseded for Windows 10. If, even as late as 2016, you can study only one version to discover what XML Microsoft uses for preparing SDB files, then version 2.1.0.3 must be it.

That said, the hope of being able to de-compile SDB files from Windows Vista up to and including Windows 8.1 such that all survive the round trip when re-compiled using the version 2.1.0.3 compiler from the one ACT turned out to be frustrated by the presence of TAG_BLOCK_UPGRADE (0x1007 in versions 6.0 and 6.1, but 0x7029 later) in the SDB files from before Windows 8.1. Though the SDB files from earlier versions record 2.1.0.3 as the compiler version, there have evidently been different versions of that compiler version!

Defaults

Some tags in the SDB file can originate in the XML but need not. Especially notable is TAG_MATCH_MODE (0x3001). The data for this tag can be 0, 1 or 2, originating in the XML as NORMAL, EXCLUSIVE and ADDITIVE, respectively. The last is the compiler’s default and I suspect that for the SDB file whose de-compilation is presented here, TAG_MATCH_MODE only ever appears as a compiler-generated default. Something similar applies to child tags of TAG_APPHELP (0x700D). Four are always written, come what may, defaulting to zero, but I doubt very much that a human preparer of the XML does actually trouble to write HTMLHELPID="0", etc.

A slightly different defaulting applies to many of the GUIDs. Some are required. Others are generated by default. When de-compiling an SDB file, it’s hard to know whether a GUID was specified in the original XML, or not, and it arguably doesn’t matter: if re-compilation is to reproduce the SDB, then all the GUIDs in the SDB file must be specified in the XML, whatever their origin.

Though the <KSHIM> and <KFLAG> tags under <KDRIVER> and <KDEVICE> end up as TAG_KSHIM_REF and TAG_FLAG in the SDB file, they actually are just references to separate <KSHIM> and <KFLAG> tags under a <LIBRARY> tag. What counts most in the reference is the NAME attribute, for looking in the library both to verify that the shim or flag is defined and to obtain other attributes such as the ID and FILE. To have these other attributes in the reference is redundant for compilation, and I treat them as if their presence in the SDB file is only by default.

Simplified Forms

Some SDB tags can originate from the XML in multiple ways. A notable example in the driver database is a TAG_DATA (0x700F) whose three child tags are TAG_NAME with “Policy” as its data, TAG_DATA_VALUETYPE (0x4018) with data REG_DWORD (4) and TAG_DATA_DWORD (0x4019) with data 1. This set of SDB tags can be explicit in the XML as 

<DATA NAME="Policy" VALUETYPE="DWORD" VALUE="1">

but can be created by the compiler as its intepretation of 

CRITICAL="NO"

in a containing <EXE> or <SYS> tag. I incline to think that the simplified attribute was introduced precisely because Microsoft meant it to be used.

Untranslated XML Tags and Attributes

More troubling for editorial discretion is that the compiler requires some tags in the XML file but does not write even a rough equivalent into the SDB file.

In the easy cases, the XML tag exists for the compiler. Specially notable is the RUNTIME_PLATFORM attribute. This is required in each <EXE> tag (and in others). The one XML file can contain <EXE> tags for any combination of the supported platforms (X86, WOW64, AMD64 and ARM). For each <EXE> in the XML file, the compiler writes a TAG_EXE in the SDB file only if the compiler’s target platform—specified by a command-line option—is among the platforms that are listed in the <EXE> tag’s RUNTIME_PLATFORM, but the latter does not itself translate to an SDB tag, e.g., as a TAG_RUNTIME_PLATFORM in the TAG_EXE. It exists solely to direct the compiler selection of <EXE> tags that are to contribute to the SDB file.

The difficult cases bring significant loss of information. As noted above, every <KFLAG> under a <KDEVICE> or <KDRIVER>, themselves under an <APP> (or some alias), must be matched to a <KFLAG> under a <LIBRARY>. This appears to be a compile-time defence against setting flags that aren’t defined. This library of defined flags, unlike the library of defined shims, is not checked at run-time and is not kept in the SDB file. The NAME that survives in the SDB file as data for a TAG_NAME under a TAG_FLAG (0x7013) is from the CLASS attribute of the <KFLAG> in the <LIBRARY>. The NAME in each <KFLAG> is therefore not knowable from the SDB file. A name might be inferred from elsewhere, but at the price of not representing only the contents of the SDB file. To escape this choice and the work of tracking down where Microsoft may have documented such names, I auto-generate a name just for the purposes of de-compilation and re-compilation—but there is certainly more to think through on this point.

See anyway that the <KFLAG> tags under <LIBRARY> cannot plausibly be complete no matter what NAME attributes are confected for them. Notably, each such tag almost certainly has a DESCRIPTION_RC_ID attribute in Microsoft’s original. Unless the compiler is run with its command-line option -r, it not only looks for this attribute but requires it if run with -u, and checks that the attributes that are present have no duplicate values. But since these values are not written into the SDB file, they are not recoverable by any amount of inspection.

Data Formats

Many of the supported XML attributes allow some freedom of expression, e.g., for choosing decimal or hexadecimal, with or without leading zeroes. Quirks abound, however. For instance, the value of an HTMLHELPID must be decimal (else it gets interpreted as zero). Dates and times require special mention. The compiler interprets them according to the current user’s defaults, e.g., as set through the Control Panel. The XML file presented above assumes U.S. formatting.
